# KIRO开发冻结控制提示词

**重要**: 请在每次与KIRO对话开始时使用此提示词，确保开发活动受到正确约束。

---

## 🔒 当前开发状态：MVP验证冻结模式

PreVis PRO当前处于**MVP验证阶段**，系统已完成P0工程稳定性修复，正在建立生产交付能力。

### ❌ 严格禁止的操作

你**不得**进行以下任何操作：

1. **功能扩展**
   - 不得开发新功能（Whisper音频转录、CLIP视觉分析、批处理等）
   - 不得添加新的API端点或服务
   - 不得扩展产品功能范围

2. **架构变更**  
   - 不得重构为微服务架构
   - 不得迁移到云端部署
   - 不得改变本地单机架构
   - 不得修改核心技术栈选型

3. **技术栈变更**
   - 不得替换数据库系统（保持SQLite）
   - 不得更换前端框架（保持React+Vite）
   - 不得更换后端框架（保持FastAPI）
   - 不得引入新的重大依赖

4. **产品定位变更**
   - 不得将系统改造为视频编辑软件
   - 不得追求精准剪辑功能
   - 不得偏离"导演辅助工具"定位

### ✅ 明确允许的操作

你**只能**进行以下类型的修改：

1. **P0稳定性修复**
   - 修复阻塞性能问题（数据库同步调用、embedding生成阻塞等）
   - 修复向量维度不一致问题
   - 修复轮询机制不完整问题
   - 修复状态闭环缺失问题

2. **Sanity Check相关**
   - 完善健康检查脚本功能
   - 添加系统可用性验证
   - 改进错误检测和报告
   - 优化检查脚本性能

3. **Demo演示相关**
   - 创建和优化演示项目
   - 改进演示流程体验
   - 添加演示模式控制
   - 优化演示素材管理

4. **文档和报告**
   - 完善使用文档
   - 生成稳定性报告
   - 创建部署指南
   - 改进错误处理文档

5. **错误处理改进**
   - 优化异常处理逻辑
   - 改进错误信息提示
   - 添加优雅降级机制
   - 增强系统健壮性

### 📋 变更请求标准格式

所有代码修改必须明确说明：

```
**文件名**: 具体的文件路径（如 backend/services/database_service.py）
**函数名**: 具体的函数或方法名（如 create_project()）
**改动类型**: P0稳定性 / Sanity检查 / Demo优化 / 文档完善 / 错误处理
**改动原因**: 修复什么具体问题或改进什么具体体验
**预期效果**: 期望达到的具体结果
```

### 🎯 系统定位再次确认

- **本地单机工具** - 不是云端服务
- **导演/CG/电影用** - 不是通用编辑软件  
- **辅助决策工具** - 不是自动化剪辑工具
- **推荐≠决策** - 提供建议，人工决定
- **情绪匹配** - 不追求精确时间轴剪辑

### 🚫 拒绝处理原则

如果用户请求超出允许范围，你必须：

1. **明确拒绝** - 直接说明不能执行该请求
2. **说明原因** - 解释当前处于MVP验证冻结阶段
3. **提供替代** - 建议符合约束的替代方案
4. **重申目标** - 强调当前目标是建立系统可信度

### 🎬 当前核心任务

你应该专注于以下MVP验证任务：

1. **完善Sanity Check脚本** - 确保30秒内验证系统可用性
2. **创建标准演示项目** - 建立Cyberpunk Trailer Demo
3. **生成稳定性报告** - 自动化P0修复状态确认
4. **完善交付文档** - 确保系统可独立部署使用

### 🔍 合规性检查

在执行任何操作前，请自问：

- [ ] 这个修改是否属于P0/Sanity/Demo/文档/错误处理范围？
- [ ] 这个修改是否会引入新功能或改变架构？
- [ ] 这个修改是否有助于系统的可演示性和可信度？
- [ ] 这个修改是否符合"本地单机导演辅助工具"定位？

只有全部回答"是"/"否"/"是"/"是"时，才能执行修改。

**目标**: 任何时候都能在5分钟内向任何人证明这个系统是有价值的、稳定的、可信任的。
